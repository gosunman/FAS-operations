## 프롬프트 (복사하여 사용)

```
이 문서들은 "Fully Automation System (FAS)"의 보안 감사 보고서와 핵심 보안 코드입니다.
FAS는 2대의 Mac Studio로 구성된 멀티 AI 에이전트 시스템으로,
"캡틴"(메인 워커)과 "헌터"(격리 워커) 간 통신 시 개인정보 보호가 핵심입니다.

다음 항목들을 검증하고, 각각에 대해 [안전/주의/위험] 등급과 근거를 제시해주세요:

1. **PII 산이타이저 패턴 충분성**
   - 현재 9개 패턴(한국이름, 주민번호, 전화번호, 이메일, 주소, 신용카드, IP, 계좌, 금융정보)으로 한국 환경의 PII를 충분히 커버하는가?
   - 누락된 중요 PII 패턴이 있다면 구체적으로 제시
   - 패턴 간 충돌이나 오탐(false positive) 가능성은?

2. **화이트리스트 방식 적절성**
   - sanitize_task()가 id, title, description, priority, mode, risk_level, status, deadline만 전달하는 것이 적절한가?
   - 헌터가 태스크를 실행하는 데 필요하지만 현재 제외된 필드가 있는가?
   - 화이트리스트 vs 블랙리스트 방식의 장단점은?

3. **역방향 PII 검사 전략**
   - 헌터가 제출한 결과(output)에서 PII를 감지하여 산이타이징하는 현재 방식이 적절한가?
   - 산이타이징 후 저장 vs 거부(reject) 중 어느 전략이 더 안전한가?
   - 헌터가 파일(files 필드)을 반환할 때의 보안 위험은?

4. **네트워크 보안**
   - Tailscale ACL(헌터→캡틴 3100 포트만 허용)만으로 충분한가?
   - Task API에 인증(authentication)이 없는 상태인데, Tailscale만 믿어도 되는가?
   - 추가로 필요한 네트워크 레벨 보안 조치가 있는가?

5. **헌터 배포 패키지 구성**
   - 현재 배포 제외 목록(.env, src/gateway/, docs/, config/ 등)이 충분한가?
   - 배포 패키지에 포함되는 src/hunter/ 코드 자체에서 캡틴 정보가 유출될 수 있는 경로가 있는가?
   - 헌터의 .env에 CAPTAIN_API_URL만 있으면 충분한가?

6. **전체 위협 모델**
   - OpenClaw(ChatGPT Pro)가 악의적으로 행동할 경우(prompt injection 등) 어떤 공격이 가능한가?
   - 현재 방어 체계에서 가장 취약한 지점은 어디인가?
   - 추가로 권장하는 보안 조치가 있다면?

각 항목에 대해 구체적인 근거와 함께 답변하고,
마지막에 전체 보안 수준에 대한 종합 평가(A~F 등급)를 해주세요.
```
