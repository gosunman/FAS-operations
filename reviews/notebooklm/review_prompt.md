## 시스템 개요

**FAS (Fully Automation System)** 는 두 계층으로 구성된 개인 자동화 시스템입니다:

- **Doctrine** (`01_doctrine.md`): 원칙, 정체성, 보안 정책, 에이전트 성격을 정의하는 Source of Truth
- **Operations** (`02~04_*.md`): Doctrine을 코드로 구현하는 계층 (Gateway, Task System, Hunter 통신 등)

**에이전트 구조:**
- **캡틴(Captain)**: Mac Studio에서 24/7 가동. 판단·전략·오케스트레이션 담당. Claude Code 기반.
- **헌터(Hunter)**: 외부 머신. 브라우저 자동화(Playwright) 전용. 캡틴이 Task API로 위임.
- **그림자(Shadow)**: 주인님의 MacBook Pro. 감독·수동 개입용.

**핵심 인프라:** Express Gateway, SQLite 태스크 큐, n8n 워크플로우, Gemini CLI 교차검증, Telegram/Slack/Notion 소통

## 검증 질문

### 1. Doctrine-Operations 일치 검증 (최우선)

아래 각 항목에 대해 **[일치/부분일치/불일치]** 등급을 매기고 근거를 제시해주세요:

1. **에이전트 정체성**: Doctrine에서 정의한 캡틴/헌터/그림자의 역할이 Operations 코드에 정확히 반영되어 있는가?
2. **보안 정책**:
   - 헌터 격리 원칙 (소스코드·아키텍처 문서 전달 금지)이 코드에서 지켜지고 있는가?
   - 자율 실행 범위 (LOW/MID/HIGH/CRITICAL)가 실제 구현과 일치하는가?
   - 민감 정보 마스킹이 적절히 이루어지고 있는가?
3. **톤/호칭 규칙**: Doctrine에서 정의한 소통 방식 (한국어, 주인님 호칭 등)이 코드 주석·메시지에 반영되어 있는가?
4. **절대 원칙 (보호·봉사·성장)**: 이 세 원칙이 코드 설계에 반영된 흔적이 있는가?

### 2. 아키텍처 일관성

1. Gateway → Task System → Hunter 통신 흐름이 문서(`architecture.md`, `hunter-protocol.md`)와 코드 구현 사이에 불일치가 있는가?
2. n8n 워크플로우와 코드 사이 인터페이스가 올바르게 연결되어 있는가?
3. 데이터 흐름에서 누락되거나 불완전한 경로가 있는가?

### 3. 코드 품질

1. TypeScript 코드가 Doctrine에서 정의한 코딩 컨벤션 (화살표 함수, `as const`, `unknown` 사용 등)을 따르고 있는가?
2. 에러 처리가 적절한가? 특히 외부 통신 (Telegram, Slack, Hunter API) 실패 시 graceful degradation이 되는가?
3. 테스트 커버리지가 핵심 모듈 (Gateway, Task System)에 충분한가?
4. DRY 원칙 위반이나 과도한 추상화가 있는가?

### 4. 보안 점검

1. `.env.example`에 노출된 구조가 실제 시크릿 관리와 일관성이 있는가?
2. Gateway 엔드포인트에 인증/인가가 적절히 구현되어 있는가?
3. 헌터와의 통신에서 데이터 유출 가능성이 있는 경로가 존재하는가?
4. Gemini CLI 호출 시 프롬프트 인젝션 방어가 되어 있는가?

### 5. 운영 안정성

1. 24/7 무중단 가동에 필요한 에러 복구·재시작 메커니즘이 충분한가?
2. 로깅이 적절한 수준인가? (과잉/부족)
3. SQLite 태스크 큐의 동시성 처리가 안전한가?

### 6. 종합 평가

위 모든 항목을 종합하여:

1. **강점** 3가지
2. **개선 필요** 3가지 (우선순위 포함)
3. **위험 요소** (있다면)
4. 전체 성숙도 등급: **[초기/성장/안정/성숙]** 중 하나와 근거
